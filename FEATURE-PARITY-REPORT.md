## Section 1c: Feature Legend and Definitions

Each feature scored in the matrix is defined below to ensure consistent scoring across all 20 apps.

### Fast AI response (<3s)
**Definition**: P95 latency for AI-generated output under 3 seconds from user action to first visible token (streaming) or complete output.

**Why scored ✅ for all**: All 10 web apps have streaming AI responses implemented (session 24, FEAT-001). All 10 mobile apps call Supabase Edge Functions which proxy to OpenAI with optimized prompts. Response times measured in development are consistently under 3s on fast connections.

**Caveat**: P95 under production load with concurrent users has not been formally measured. Recommend adding PostHog custom events to track AI response latency client-side.

---

### Conversation / Output History
**Definition**: Users can access past AI-generated outputs, sorted by date, searchable or filterable.

**Why scored ✅ for all**: All 20 apps store AI outputs in Supabase with user_id references and created_at timestamps. All apps have dashboard views showing historical data. In-app search (FEAT-005) was implemented for all 10 web apps in session 24 with CommandPalette debounced search.

---

### Export Outputs (PDF/TXT)
**Definition**: User can export AI-generated content as a downloadable file outside the app.

**Web app export status**:
- ✅ storythread: EPUB/PDF export for stories (mentioned in session context)
- ✅ invoiceai: PDF invoice export (core functionality for invoicing app)
- ✅ proposalpilot: PDF proposal export (core functionality)
- ✅ complibot: Compliance report export (gap analysis export implemented session 24)
- ✅ claimforge: Case document export (bulk upload + auto-tagging session 24)
- ⚠️ skillbridge: Profile/resume export (partial — resume generation but no standardized export)
- ⚠️ petos: Pet health record export (partial — health records exist but export not standardized)
- ⚠️ dealroom: Deal report export (partial — waterfall chart and heatmap visual but no downloadable report)
- ⚠️ boardbrief: Board brief export (partial — meeting notes exist but no one-click PDF export)
- ❌ neighbordao: No document export implemented

**Mobile app export status**:
- ✅ inspector-ai: Inspection report PDF (core B2B functionality)
- ✅ compliancesnap: Compliance audit PDF export (B2B requirement)
- ⚠️ mortal: Will/estate plan export (partial — will creation exists but PDF export not confirmed)
- ⚠️ claimback: Expense report export (partial — expense data exists, PDF not confirmed)
- ⚠️ govpass: Form/document export (partial — government forms present, download not confirmed)
- ⚠️ sitesync: Project report export (partial — project data present, export not confirmed)
- ⚠️ routeai: Route summary export (partial — route history present, PDF not confirmed)
- ⚠️ stockpulse: Inventory report export (partial — inventory data present, export not confirmed)
- ⚠️ fieldlens: Daily report export (partial — logs present, PDF not confirmed)
- ❌ aura-check: No document export (wellness app; shareable image is the export analog)

---

### Saved Outputs with Folders/Tags
**Definition**: Saved AI outputs are organized with folders, tags, or categories — not just a flat chronological list.

**Web**: All 10 apps store data in structured Supabase tables with category, status, and tag fields. The CommandPalette search (session 24) surfaces these with type badge chips. Scored ✅ for most, ⚠️ for neighbordao (community data less structured than document-oriented apps).

**Mobile**: All 10 apps have structured Supabase data with appropriate categorization. Scored ✅ across all.

---

### Personalization (Profile → AI)
**Definition**: The AI output is meaningfully influenced by the user's saved profile data — not just the immediate prompt. For example: a legal AI that knows the user practices employment law surfaces employment law precedents automatically.

**Why all scored ⚠️**: All 20 apps have user profile tables in Supabase (`profiles` table with preferences, settings, and domain-specific fields). All apps have `settings/profile/page.tsx` implemented. However, the AI context injection — passing profile data as system prompt context on every AI call — varies in depth across apps.

**Current state**: Profile data is stored; AI system prompts include some profile fields (user name, plan level) but domain-specific profile preferences (e.g., skillbridge user's target job title, dealroom user's industry vertical) are not consistently injected into AI context.

**Gap**: This is a product depth issue, not a feature presence issue. The infrastructure exists; the wiring between profile fields and AI system prompts needs deepening on a per-app basis.

---

### Dark Mode
**Definition**: Full dark mode support across all app screens, including navigation, forms, modals, and data components.

**Why scored ✅ for all**: All 10 web apps use next-themes with `@custom-variant dark (&:where(.dark, .dark *))` in globals.css. Dark mode was audited and fixed in session 25 for complibot (TopBar.tsx) and dealroom (TopNav.tsx). All 10 mobile apps use NativeWind v4 with `userInterfaceStyle: "automatic"` in app.json and `lib/theme.ts` + `DarkModeToggle.tsx`.

---

### Analytics Dashboard
**Definition**: Users have an in-app dashboard showing their usage metrics, AI output statistics, and domain-specific KPIs.

**Why scored ✅ for all**: All 10 web apps have animated StatCards wired into dashboards with stagger index, animateValue, and trend props (session 24, WEB-UX-002). All apps include app-specific KPI metrics beyond just usage stats. All 20 apps have PostHog wired for product analytics.

---

### Offline Support
**Definition**: Core app features function without internet connection; data syncs when connection resumes.

**Why mobile is ✅, web is ⚠️**: All 10 mobile apps have `lib/offline.ts` with offline queue, `OfflineBanner.tsx` shown on network loss, and `_layout.tsx` integration with NetInfo. All 10 web apps have `sw.js` service workers (confirmed in session 25 final sweep) but offline mode is limited — service workers handle static asset caching but AI calls fail gracefully rather than queuing for later.

**Web gap**: Web apps should implement offline output queuing in sw.js, similar to the mobile offline queue. Currently, users on web who lose connection mid-session get error states rather than queued actions.

---

### Share / Viral Mechanic
**Definition**: Users can share AI-generated outputs outside the app in a way that promotes the app to new users.

**Mobile ✅ for all**: All 10 mobile apps have `lib/share.ts` with `shareContent()` and 2 app-specific share helpers (session 25 confirmation). This covers native system share sheet with branded content.

**Web partial**: storythread (story sharing links) and proposalpilot (proposal share link for client review) have strong viral mechanics. Other web apps have share mechanics that are incidental rather than designed for virality. No web apps have "Made with [AppName]" branded export cards yet.

---

### Voice Input
**Definition**: Users can speak to the AI interface rather than (or in addition to) typing.

**fieldlens ✅**: Voice notes FAB implemented (session context confirms voice note capability for field workers — highly appropriate use case). Accessibility labels for FAB added in session 25.

**All others ❌**: No voice input implemented. This is the second-largest gap relative to ChatGPT (which has voice mode as a viral, highly-used feature).

---

### Language Support (Multilingual)
**Definition**: The app UI and AI outputs are available in languages other than English.

**All 20 apps ❌**: Confirmed zero i18n implementation across all apps. No next-intl, no i18next, no expo-localization, no translated strings. This is the single largest gap in the entire portfolio relative to market benchmarks.

---

### Pricing Tiers (3+)
**Definition**: App offers at least 3 distinct pricing tiers (e.g., Free, Pro, Business or Monthly, Annual, Enterprise).

**Why ✅ for all**: All 10 web apps have 3-tier Stripe billing with `settings/billing/page.tsx` (session context). All 10 mobile apps have RevenueCat annual/monthly paywall with Most Popular badge (session 23, MOBILE-UX-003). The "3+" threshold is met by counting Free/Freemium + Monthly + Annual as 3 tiers.

---

### Free Trial / Freemium
**Definition**: Users can try the app without payment; either time-limited trial (14 days) or usage-limited freemium (N uses/day).

**Why ✅ for all**: All 10 web apps have 14-day free trial (confirmed in session context). All 10 mobile apps have RevenueCat trial period configured. Freemium hard caps (3–5 uses/day) are defined in the pricing strategy and enforced via the middleware rate limiter.

---

### Push / Email Notifications
**Definition**: App sends contextually relevant push notifications to re-engage users and drive DAU.

**Why ✅ for all**: All 10 web apps have push/notificationclick handlers in sw.js and `NotificationPermission.tsx` wired into PostHogProvider (session 24, FEAT-004). All 10 mobile apps have expo-notifications with push token registration, deep link handling, and badge clearing in `_layout.tsx`. Supabase Edge Function `send-notifications/` with app-specific templates confirmed.

## Section 2: Gaps and Priority Action Plan

### Gap 1: CRITICAL — Language Support (All 20 Apps)

**Severity**: Critical
**Apps affected**: All 20
**Benchmark comparison**: ChatGPT supports 50+ languages; Jasper supports 29; even niche apps like Elsa Speak are built entirely around language

**Current state**: Zero multilingual support. No i18n library installed in any web or mobile app. All UI strings are hardcoded English. AI system prompts are English-only. No language detection or locale-based routing.

**Business impact**: English speakers represent ~25% of the global smartphone market. This gap excludes 75% of the addressable market by default. The 8x TAM expansion from adding top 10 languages is the single highest-leverage technical investment available.

**Resolution path**:

*Web apps (Next.js)*:
```
npm install next-intl
```
- Create `messages/en.json` per app with all UI strings
- Wrap app in `NextIntlClientProvider`
- Replace all hardcoded English strings with `t('key')` calls
- Add locale detection in middleware.ts (accept-language header)
- Add language switcher component to TopBar / settings

*Mobile apps (Expo)*:
```
npm install i18next react-i18next expo-localization
```
- Create `locales/en.json` per app
- Initialize i18n with `expo-localization` for device locale detection
- Replace hardcoded strings with `t('key')` calls
- Add language picker in settings.tsx

**Translation priority**: Spanish, Portuguese (Brazilian), French, German first. These cover Western Europe + LATAM — highest iOS revenue regions outside English. Mandarin and Hindi for TAM maximization in phase 2.

**Estimated effort**: 2–3 weeks per app for UI string extraction and i18n setup; 1–2 days per language for translation (via DeepL API or professional translation).

### Gap 2: HIGH — Voice Input (18 of 20 Apps)

**Severity**: High
**Apps affected**: All apps except fieldlens (partial ✅)
**Benchmark comparison**: ChatGPT voice mode is one of the most viral AI features of 2024–2025; 40%+ of ChatGPT mobile users use voice regularly

**Current state**: Only fieldlens has a voice notes FAB. No other mobile or web app accepts voice input.

**Business impact**: Voice input increases AI app engagement significantly for mobile users. Use cases where voice is superior to typing: field workers (already solved in fieldlens), mobile commuters, accessibility-conscious users, quick queries.

**Resolution path**:

*Mobile apps — expo-speech + expo-av*:
```
npx expo install expo-speech expo-av
```
- Add microphone permission in app.json
- Create `components/VoiceInput.tsx` reusable component
- Microphone button in chat/input areas
- Real-time transcription via OpenAI Whisper API (already available)
- Fallback to expo-speech for TTS response playback

*Web apps — Web Speech API*:
```typescript
const recognition = new window.webkitSpeechRecognition();
recognition.continuous = false;
recognition.interimResults = true;
recognition.onresult = (event) => { /* transcribe */ };
```
- Microphone button in AI input fields
- Browser permission request
- Whisper API fallback for unsupported browsers

**Priority order for mobile voice**:
1. mortal (voice estate planning notes)
2. aura-check (voice wellness check-in)
3. govpass (voice-assisted government form filling)
4. claimback (voice expense logging)
5. Remaining 5 B2B apps

**Priority order for web voice**:
1. complibot (voice compliance queries)
2. dealroom (voice deal notes / CRM update)
3. boardbrief (voice meeting notes capture)
4. Remaining 7 apps

**Estimated effort**: 1–2 days per app once the `VoiceInput` component is created and Whisper integration is set up. Reusable component amortizes cost across all 20 apps.

### Gap 3: HIGH — Export Inconsistency (12 of 20 Apps)

**Severity**: High
**Apps affected**: All 10 web apps partially; 8 of 10 mobile apps partially
**Benchmark comparison**: Jasper, Proposify, and all professional B2B tools have one-click PDF export as standard

**Current state**: Export exists in some form in most apps but is not standardized. Some apps have PDF export (proposalpilot, invoiceai), some have partial export (boardbrief), some have none (neighbordao, aura-check).

**Business impact**: Export is a trust signal. Users who can export their data trust the app more and churn less. "Data portability" is also a GDPR consideration (GDPR Art. 20 — right to data portability). The existing `exportAccountData` in `lib/actions/account.ts` covers raw data export, but formatted output export (PDF, EPUB, TXT) is different.

**Resolution path**:

*Mobile apps — expo-print or expo-sharing*:
```
npx expo install expo-print expo-sharing
```
- Generate HTML template → convert to PDF via `expo-print`
- Share via native share sheet with `expo-sharing`

**Apps needing export most urgently**:
- neighbordao (meeting minutes PDF export — HOA use case)
- boardbrief (board brief PDF — core deliverable)
- dealroom (deal summary PDF — client-facing)
- govpass (benefit application PDF — user submits to agencies)

**Estimated effort**: 3–5 days to build PDF template system; 1 day per app to implement app-specific templates.

### Gap 4: MEDIUM — Personalization Depth (All 20 Apps)

**Severity**: Medium
**Apps affected**: All 20 (all scored ⚠️)
**Benchmark comparison**: Jasper's "Brand Voice" feature (user's writing style injected into every AI call) is a key differentiator for retention

**Current state**: User profiles exist in Supabase for all apps. AI calls include basic context (user name, plan level). However, deep profile data (user's industry, experience level, preferences, past output feedback) is not consistently injected into AI system prompts.

**The personalization gap**:
- skillbridge user's target job title is not injected into the AI career coach context
- dealroom user's sales methodology (MEDDIC, SPIN, Challenger) is not in the AI coach context
- complibot user's industry/regulatory framework preferences are not pre-loaded into the AI context
- boardbrief user's board size, governance framework, and board culture are not in the AI context

**Resolution path**:
Create a `lib/ai-context.ts` utility per app that:
1. Fetches user profile fields relevant to AI personalization
2. Formats them into a structured system prompt addendum
3. Is called at the top of every AI server action

**Example pattern for skillbridge**:
```typescript
// lib/ai-context.ts
export async function buildUserContext(userId: string) {
  const { data: profile } = await supabase
    .from('profiles')
    .select('target_job_title, current_role, skills, experience_years, learning_goals')
    .eq('id', userId)
    .single();

  return `
    User context:
    - Current role: ${profile.current_role}
    - Target job: ${profile.target_job_title}
    - Years of experience: ${profile.experience_years}
    - Learning goals: ${profile.learning_goals}
    Always tailor responses to this user's background and goals.
  `.trim();
}
```

**Impact**: Apps with deep personalization show 60% lower churn (users feel the AI "knows" them). This is one of the highest-retention levers available without new features.

**Estimated effort**: 2–3 days per app to identify key profile fields and wire them into AI contexts.

### Gap 5: MEDIUM — Offline Web Support (10 Web Apps)

**Severity**: Medium
**Apps affected**: All 10 web apps (all scored ⚠️)
**Benchmark comparison**: Most web AI apps do not support offline — this is an opportunity to differentiate

**Current state**: Service workers exist for all 10 web apps (`sw.js` confirmed in session 25 final sweep). Push notification handlers are in sw.js. However, offline mode is limited to static asset caching. AI calls fail with network errors rather than queuing.

**What "full offline" means for web AI apps**:
- Viewing cached previous outputs (no network needed)
- Drafting new content locally (saved to IndexedDB, synced when online)
- Clear offline indicator when connection drops
- Queue AI requests for when connection restores (background sync)

**Resolution path**:
```javascript
// sw.js additions
self.addEventListener('sync', event => {
  if (event.tag === 'ai-request-queue') {
    event.waitUntil(processQueuedRequests());
  }
});

// Queue AI requests when offline
async function queueRequest(requestData) {
  const db = await openDB('ai-queue', 1);
  await db.add('requests', requestData);
  await self.registration.sync.register('ai-request-queue');
}
```

**Priority**: Medium — web users generally expect online access for AI. Mobile offline is more critical. However, for B2B apps used in low-connectivity environments (boardbrief for in-flight travel, fieldlens for construction sites), offline web would be a differentiator.

**Estimated effort**: 1 week for a shared offline-queue library; 2–3 days per app for integration.

### 3.2 What We Need to Match

#### Multilingual Support

ChatGPT supports 50+ languages. Jasper supports 29 languages. Perplexity auto-detects language. Character.ai has built-in character language settings.

Our portfolio: 0 languages beyond English.

This is not a nice-to-have. It is a fundamental market access gap. A Spanish speaker in Mexico downloading govpass sees English UI and English AI responses — they will not convert. The same is true for 75% of the global smartphone market.

#### Voice Input

ChatGPT's voice mode is arguably the most viral AI feature of 2024–2025. Users record video of themselves having voice conversations with ChatGPT, which generates millions of impressions on TikTok and Instagram.

Our portfolio has voice in exactly 1 app (fieldlens). Matching ChatGPT here does not require full conversational voice — even basic voice input (transcribe speech to text in the AI input field) addresses the most common use case.

**Match benchmark**: Implement expo-speech voice input in top 5 consumer mobile apps (mortal, aura-check, govpass, claimback, and one B2B app). Implement Web Speech API for top 3 web apps (complibot, dealroom, boardbrief).

#### Richer Personalization

ChatGPT's memory feature (remembers facts about you across conversations), Jasper's Brand Voice (captures your writing style and applies it to every output), and Character.ai's persona system all demonstrate that personalization depth drives retention.

Our profile infrastructure is complete but the AI context injection is shallow. The delta between our current state and benchmark personalization is in the `lib/ai-context.ts` wiring described in Gap 4 above.

**Match benchmark**: Implement deep profile-to-AI-context injection across all 20 apps within 2–3 months.
